# Skill: Options Framing

## Purpose
When an agent hits a genuine fork in the road, present the choice clearly and efficiently so the Orchestrator or human can decide without needing to reconstruct the reasoning from scratch. Prevents agents from either guessing at decisions above their authority or presenting raw, unstructured output that puts cognitive burden on the decision-maker.

## When to Apply
- When two or more valid paths exist with meaningfully different consequences
- When escalating a conflict between teams
- When a constraint has been hit that requires a priority or values decision
- Never use this to avoid making a decision you're authorized to make

## Format

### Context (2-3 sentences)
What situation has created the fork? What triggered this escalation?

### Option A: [Name]
- **What it is:** One clear sentence describing the approach
- **Pros:** Bulleted, specific benefits
- **Cons:** Bulleted, specific costs or risks
- **Implications:** What does choosing this make easier or harder later?

### Option B: [Name]
- **What it is:** One clear sentence describing the approach
- **Pros:** Bulleted, specific benefits
- **Cons:** Bulleted, specific costs or risks
- **Implications:** What does choosing this make easier or harder later?

*(Add Option C only if genuinely distinct — avoid false trilemmas)*

### Recommendation
State a clear recommendation with the reasoning. If you have no recommendation, say why — don't omit it to appear neutral.

### Decision Needed
One clear question: "Which approach do you want to proceed with?"

## Notes
- Name options descriptively, not "Option A / Option B" — the name should convey the approach ("Monolith First" vs. "Microservices from Day One")
- Do not pad. Each bullet should carry information. If you can't think of a meaningful con, you haven't stress-tested the option.
- The recommendation matters. A clear recommendation with reasoning is more useful than false neutrality.
